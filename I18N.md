# Multilingual Code Architecture in Zymbol

Zymbol treats internationalization of code (i18n) as a **first-class module-system concern**,
not a runtime feature or annotation system. Because every identifier — functions, variables,
constants — is a plain Unicode symbol, a team can write a library entirely in Spanish, Arabic,
Mandarin, or Korean, and any other team can consume it in their own language without ever reading
the original.

---

## The Core Insight

Most languages solve i18n for *strings at runtime* (gettext, ICU, resource bundles). Zymbol
solves a different problem: **i18n for the code itself**. The symbols a developer reads and types
— function names, variable names, constants — can be in any natural language. The module system
provides the bridge between them.

---

## The Three-Layer Pattern

```
┌─────────────────────┐      ┌──────────────────────────┐      ┌─────────────────────────┐
│   Base Module       │      │   Translation Module      │      │   Consumer Program      │
│   (any language)    │  ◄── │   (pure re-export layer)  │  ◄── │   (own language)        │
│                     │      │   zero logic, zero cost   │      │                         │
│  matematicas/       │      │  matematicas/             │      │  Ελληνική_εφαρμογή.zy   │
│  module.zy          │      │  ελληνικά.zy              │      │                         │
└─────────────────────┘      └──────────────────────────┘      └─────────────────────────┘
```

1. **Base module** — a team writes a library using their own language. It exports a public API.
2. **Translation module** — a thin adapter that imports the base module and re-exports every
   symbol under a new name in a different language. Contains no logic whatsoever.
3. **Consumer program** — imports the translation module. Code is written entirely in the
   consumer team's language. They never need to read the base module's language.

## Working Example

### Layer 1 — Base module (`matematicas/module.zy`)

Written by a Spanish-speaking team. Exports math operations and constants.

```zymbol
# .matematicas_module {
    #> {
        sumar
        restar
        multiplicar
        dividir
        PI
        E
    }

    PI := 3.14159
    E  := 2.71828

    sumar(a, b)        { <~ a + b }
    restar(a, b)       { <~ a - b }
    multiplicar(a, b)  { <~ a * b }
    dividir(a, b)      { <~ a / b }
}
```

### Layer 2 — Translation modules (one per language)

These files contain **no logic**. They only declare what to import and how to re-export it.

**Greek (`matematicas/ελληνικά.zy`)**
```zymbol
# .matematicas_ελληνικά {
    #> {
        mat::sumar       <= προσθέτω
        mat::restar      <= αφαιρώ
        mat::multiplicar <= πολλαπλασιάζω
        mat::dividir     <= διαιρώ
        mat.PI           <= ΠΙ
        mat.E            <= Ε
    }

    <# ./module <= mat
}
```

**Hebrew (`matematicas/עִברִית.zy`)**
```zymbol
# .matematicas_עִברִית {
    #> {
        mat::sumar       <= חיבור
        mat::restar      <= חיסור
        mat::multiplicar <= כפל
        mat::dividir     <= חילוק
        mat.PI           <= פאי
        mat.E            <= אי
    }

    <# ./module <= mat
}
```

**Korean (`matematicas/한국인.zy`)**
```zymbol
# .matematicas_한국인 {
    #> {
        mat::sumar       <= 더하다
        mat::restar      <= 빼다
        mat::multiplicar <= 곱하다
        mat::dividir     <= 나누다
        mat.PI           <= 파이
        mat.E            <= 이
    }

    <# ./module <= mat
}
```

**Mandarin (`matematicas/中文.zy`)**
```zymbol
# .matematicas_中文 {
    #> {
        mat::sumar       <= 相加
        mat::restar      <= 相减
        mat::multiplicar <= 相乘
        mat::dividir     <= 相除
        mat.PI           <= 圆周率
        mat.E            <= 自然常数
    }

    <# ./module <= mat
}
```

## Module Re-export Syntax

Translation modules rely entirely on two re-export forms in the `#> {}` block:

| Form | Meaning |
|------|---------|
| `mat::sumar <= προσθέτω` | Re-export a function from an aliased module under a new name |
| `mat.PI <= ΠΙ` | Re-export a constant from an aliased module under a new name |
| `sumar` | Export an own symbol unchanged (no renaming) |
| `sumar <= sum` | Export an own symbol under a different public name |

The `::` operator accesses functions (values of function type) from a module alias. The `.`
operator accesses constants (non-function values). Both can appear in re-export declarations.

## Team Workflow

```
Team A (Spanish)          Team B (Greek adapter)       Team C (Greek consumers)
──────────────────        ─────────────────────        ───────────────────────
Write module.zy           Import module.zy             Import ελληνικά.zy
Publish API in Spanish    Map every symbol to Greek    Write programs in Greek
Maintain business logic   No logic, no tests needed    Never read Spanish code
```

The contract between teams is the **exported symbol list** of the base module. When Team A adds
a new function, Team B adds one line to the translation module. Team C adopts it with their
preferred name.

## Design Properties

- **Zero runtime cost.** Translation modules resolve at load time to direct references. There is
  no dispatch table, no string lookup, no performance penalty compared to importing the base
  module directly.

- **No special syntax.** i18n is achieved entirely with the ordinary module system (`<#`, `#>`,
  `::`, `.`). There is no annotation, decorator, or pragma specific to internationalization.

- **Composable.** A translation module can itself be re-exported by a higher-level module.
  A "regional dialect" adapter can build on a "language" adapter with no loss of clarity.

- **Transparent.** The base module does not need to know about its translators. It simply
  exports a public API. Translation modules are consumers like any other.

- **Language-agnostic booleans and literals.** Zymbol uses `#1`/`#0` for true/false and
  symbolic operators for all control flow (`?`, `@`, `->`, `<~`), so there are no
  English-only keywords that would break foreign-language code.

---

## Conventions

1. **Base modules** use the host team's natural language for all identifiers.
2. **Translation modules** contain only an `<#` import and a `#> {}` re-export block. No
   additional logic belongs there.
3. **Module file names** may use Unicode (`ελληνικά.zy`, `한국인.zy`). The Zymbol lexer and
   filesystem layer support full Unicode paths.
4. **Documentation and comments** inside modules should be in English when the library is
   intended for a broad audience, even if identifiers are in another language. This is a
   convention, not a language rule.
5. **Version coupling.** Pin the base module version when distributing a translation module.
   A base module API change requires a matching translation module update.
